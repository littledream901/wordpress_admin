遵循 FastAPI + Vue 后台项目规范开发：

## 总原则
- 保持分层清晰、优先复用、接口稳定、权限统一、配置外置。
- 修改前优先检索现有 `utils`、公共组件、`services`、基础 CRUD，先复用再新增。
- 改动要保证可运行：后端接口、前端页面、数据库迁移、权限、日志链路保持一致。

## 后端规范
- `api/v1/`：只做路由注册、基础参数校验、调用 controller，禁止写核心业务逻辑、数据库复杂查询、第三方请求。
- `controllers/`：只做业务编排、CRUD 流程、参数组装，公共能力优先复用 `core` / `services` / `utils`。
- `services/`：专门封装第三方服务和跨模块能力（Cloudflare、Dynadot、Shopify、Gmail、HubStudio、Woo、1Panel 等），统一超时、重试、异常处理。
- `models/`：只定义 Tortoise ORM 表结构、索引、唯一约束、关联关系，不写业务逻辑。
- `schemas/`：只定义 Pydantic 请求/响应模型，区分 `Create / Update / Response`，保证字段类型完整。
- `utils/`：只放可复用纯函数，如格式化、脱敏、校验、导出、日期处理、加解密等。
- `settings/`：统一从环境变量读取配置，禁止硬编码密钥、Token、URL、账号密码。
- `core/`：认证、RBAC、异常、中间件、日志、基础 CRUD 属于基础设施，优先继承和扩展，不轻易直接改源码。
- `agent/`：独立进程逻辑与主服务隔离，不混写到 controller/service 主链路。
- `migrations/`：数据库迁移统一用 Aerich 生成，不手改迁移文件。

## 后端硬规则
- 新增接口默认接入 RBAC 权限校验。
- 关键操作默认增加审计日志，记录用户、IP、操作对象、结果。
- 外部服务请求必须统一加：超时、重试、错误归一化、日志脱敏。
- 列表接口统一返回分页结构，前后端字段保持一致。
- 初始化逻辑必须幂等，不能重复建表、重复插入种子、重复修复脏数据。
- 数据修复脚本、种子脚本、迁移脚本必须分离，不能全部塞进启动入口。
- 新增数据库表或字段时，必须同步补模型、Schema、迁移、接口、前端类型。
- 数据库层必须补唯一约束和必要索引，不能只靠代码防重复。
- 角色、权限、管理员判断统一使用业务 `code` 或明确唯一键，禁止依赖中文名。
- 核心模型必须显式从具体文件导入，禁止从 `app.models` 聚合入口导入。
- 禁止 `import *` 用于业务代码、模型聚合、权限链路。

## 前端规范
- `components/common/`：放公共组件（表格、表单、弹窗、搜索栏、状态标签等），要求可配置、无业务硬编码。
- `components/business/`：放业务模块组件，与页面解耦。
- `api/`：统一 axios 请求封装，按模块拆分，禁止页面直接裸发请求。
- `views/`：只负责页面渲染、事件绑定、调用 api/store，复杂逻辑抽到 hooks/composables。
- `router/`：动态路由基于后端菜单/权限生成，不随意硬编码业务路由。
- `store/`：统一管理用户信息、权限、全局配置、页面共享状态。
- `utils/`：放前端格式化、校验、导出等通用逻辑。

## 前端硬规则
- 不复制通用表单、弹窗、表格模板，优先复用公共组件。
- 不硬编码接口地址、Token、第三方密钥、权限判断。
- 不在页面模板或单文件组件中堆大量通用格式化/校验逻辑。
- 前端权限逻辑统一从 store / 后端菜单接口获取，不在页面里散写。
- 公共组件避免全局样式污染，业务样式局部收口。

## 容易踩坑（重点避免）
- 不要从 `app.models` 聚合入口导入核心模型，统一从 `app.models.admin` / `app.models.xxx` 显式导入。
- 不要使用 `import *`，容易造成同名模型、同名工具函数覆盖。
- 不要在启动流程里做高风险 DDL/DML（如修主键、删重、重建数据），这类逻辑必须独立成 repair 脚本。
- 不要把迁移、种子、修复混在一个入口执行。
- 不要在 controller 里直接写第三方接口调用、重试、异常处理。
- 不要让前端和后端字段名、分页结构、状态值各写一套。
- 不要只在代码层防重复，唯一约束必须落到数据库。
- 不要依赖中文角色名、菜单名做权限判断，统一用 `code` / 唯一键。
- 不要在页面里直接拼权限判断、直接请求接口、直接处理复杂业务流程。

## 新增功能顺序
- 后端：先定 model / schema，再写 service / utils，再写 controller，再接 api 路由。
- 前端：先定 api/types，再写 hooks/composables，再接 view/components。
- 涉及数据库变更：先改模型，再生成 Aerich 迁移，再联调接口和前端。
- 涉及权限：默认补 RBAC、菜单/API 权限、审计日志。

## 默认质量要求
- 保持类型完整：Python/Pydantic/TS 类型尽量明确。
- 函数命名、字段命名、返回结构统一风格。
- 公共逻辑抽离，避免复制粘贴。
- 修改完成后至少保证：能启动、接口可用、迁移可执行、前端页面能联调。

## 任务执行指令
- 重构代码：遵循本规范重构，优先复用公共组件、utils、services，避免重复实现。
- 新增页面：按现有后台模板结构创建页面，接入对应 v1 接口，复用公共表单/表格组件。
- 新增接口：按 v1 模块结构创建路由、schema、controller，补 RBAC、审计日志、类型定义和迁移。
